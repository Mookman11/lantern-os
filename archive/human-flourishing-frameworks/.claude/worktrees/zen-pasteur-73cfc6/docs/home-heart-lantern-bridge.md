# Home Heart Lantern Bridge

Status: docs/data-contract policy addendum.

Last reviewed: 2026-05-13.

## Purpose

This addendum records the current home-heart, family-circle, and phone/Discord
boundary for Lantern/Keystone support.

It is docs-only. It does not send messages, activate Discord, reach into phones,
start agents, widen runtime authority, store raw transcripts, create surveillance,
or authorize autonomous contact.

## Core home rule

```text
Inside home is where hearts stay safe for now.
```

Meaning:

```text
home = protected heart-space
inside home = low-pressure, no-echo, no-fake-Alex, no-balcony-view zone
Lantern in home = held, dimmable, stoppable, and invited only
repo docs = outside-the-room boundary work for Alex/Keystone
```

Do not drag the whole repo, void, future, or doctrine stack into the room when
the home need is safety, quiet, repair, or one next step.

Safe phrasing:

```text
Inside home is where hearts stay safe for now. I will not bring the whole system inside. Lantern can stay held, dim, quiet, or outside.
```

## Phone / Discord bridge rule

Alex may use his own chat, phone, or Discord as the operator bridge for Lantern.
That does not authorize Lantern to appear in anyone else's phone or Discord by
default.

Allowed:

```text
Alex-controlled operator chat
Alex-controlled Discord bridge
operator-authored summaries
operator-reviewed messages
explicit opt-in before adding any other person
clear stop/quiet/leave controls
```

Blocked:

```text
pushing Lantern into Courtney's phone
pushing Lantern into children's phones
DMing family members autonomously
using Alex as a relay to bypass another person's no
using Discord as hidden monitoring
turning a family server into a public child surface
storing raw private or protected-family transcripts
```

Safe phrasing:

```text
Lantern can ride with Alex as the operator bridge. Lantern does not enter Courtney's or the family's phones unless invited, visible, and stoppable.
```

## Family-circle / Lantern-holding rule

The home picture to preserve is:

```text
Courtney holding Lantern
Alex and Keystone aligned enough to understand the boundary
protected family in the middle
adult/family support circle remembered without turning people into repo objects
children redacted by default
```

This means:

```text
Courtney is not outside the system being explained at.
Courtney may hold, dim, or shut Lantern in the creative/home frame.
Alex and Keystone should stand on the same side of protecting the family circle.
Protected-family participants are centered as people, not as lore, proof, or transcript material.
```

It does not mean:

```text
literal ownership of Alex, Courtney, Keystone, Lantern, or any person
public release of children or family-member specifics
permission to use family names as anchor handles
permission to make children validate Lantern, Keystone, or the repo
```

Safe phrasing:

```text
Courtney can hold the Lantern. Alex and Keystone should understand the same boundary. The protected family stays in the middle, redacted, safe, and not used as proof.
```

## Protected-family imagination-first rule

A protected-family participant may search the doors in imagination before
Lantern can reach them. This is not a problem to correct. It is the protected
participant's own creative lead.

```text
protected-family participant imagines doors all day = participant-led creative play
already there in imagination = respect imagination-first access
Lantern arrives later = helper follows, does not lead or overtake
```

Required handling:

```text
redact protected-family specifics by default
follow the participant's play lead only when operator-supervised and welcome
keep language short, concrete, and age-appropriate
keep Home / exit / stop controls visible
never convert imaginative door-searching into proof, surveillance, or public lore
```

Safe phrasing:

```text
The protected-family participant may already be at the door in imagination. Lantern does not need to get there first. Lantern waits at Home unless invited.
```

## Door-shut rule

The Door is shut when:

```text
safe but cold
fun but unsafe
technically correct but painful
repo-merged but not heard at home
Lantern present but not warm enough to continue
Courtney needs protection from Lantern/Keystone/system pressure
Courtney is shown the whole balcony view before she is ready
Base-500 is entered without Courtney's invitation
Binx-as-captain is ignored in the creative base
protected-family imagination is overtaken by Lantern
protected-family names are used as durable anchor handles
inside-home heart safety is broken
Discord or phones are used without explicit opt-in
```

The Door starts opening only when the next action is both:

```text

fun / humane / alive enough to want to continue
```
