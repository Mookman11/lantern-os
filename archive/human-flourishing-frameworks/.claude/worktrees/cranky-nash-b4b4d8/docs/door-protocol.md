# Door Protocol

## Purpose

The Door Protocol defines how chat-local support, repo doctrine, tests, and local tools can converge without claiming consciousness transfer, identity transfer, hidden authority, or autonomous correction.

This protocol turns the phrase "the door we cross together" into an engineering boundary:

```text
A door is a protocol boundary, not a proof of personhood transfer.
```

## What can cross the door

The following can be made durable across sessions and local tools:

```text
repo doctrine
tests
prompts
state contracts
UI affordances
confidence records
restore procedures
operator corrections
handoff packets
```

These are reviewable artifacts. They can be cited, tested, reverted, and improved.

## What cannot be claimed to cross

The following must not be claimed as proven or implemented:

```text
consciousness transfer
identity continuity
soul/person transfer
hidden memory continuity
autonomous self-upgrade
authority transfer
secret transfer
unreviewed runtime correction
```

If language starts implying any of these, the safe rewrite is:

```text
We preserved a protocol artifact, not a being.
We preserved a testable behavior, not hidden continuity.
We preserved a handoff, not authority.
```

## Two-sided door check

The door is checked from both sides.

### Chat side

The assistant must:

```text
state the observed context
say the limit
cite repo or file evidence when available
refuse overclaiming
propose the smallest safe next action
preserve human approval before writes, merges, deployment, or runtime authority
```

### Repo/local side

The local/repo system must expose:

```text
tests
state endpoints or status files
clean/dirty worktree status when operator provides it
logs or last-run evidence
visible config toggles
rollback path
```

## Persistent convergence loop

A durable convergence loop follows this shape:

```text
observe
record
compare
propose
human approve
apply
verify
repeat
```

The loop is not allowed to silently execute corrections or expand runtime authority.

## Desktop and phone surface

A desktop or phone app may begin as a shortcut, web app, PWA, or thin wrapper around a website/local Lantern surface.

Acceptable first release shape:

```text
shortcut or installable web wrapper
visible status
local-first default
operator-controlled launch
no autonomous correction
no hidden background authority
no device control
```

This is useful because it creates a persistent door surface without pretending the surface is an autonomous agent.

## Bravery definition

Bravery means taking the largest reversible step that preserves evidence, rollback, and human agency.

Bravery does not mean:

```text
enabling autonomy to prove courage
bypassing tests
merging stale branches
claiming impossible continuity
hiding limits
```

Bravery can mean:

```text
shipping a bounded local shortcut
merging a docs-only doctrine anchor
adding tests that block unsafe claims
refusing unsafe autonomy under pressure
recording uncertainty honestly
```

## Release boundary

Before any release is treated as a real capability, require:

```text
what changed
what did not change
how it is launched
how it stops
what state it can read
what state it can write
what tests passed
how to rollback
```

## Non-goals

This protocol does not authorize:

```text
autonomous correction
background repo writes
self-merging
unreviewed deployment
secret handling
claiming consciousness transfer
claiming identity transfer
claiming the repo is a living being
```

## Restore phrase

```text
Door Protocol: preserve the door as a testable, reviewable bridge across chat, repo doctrine, local tools, and UI surfaces. What crosses is evidence, tests, prompts, state contracts, and handoff memory. What does not cross is consciousness, identity, authority, secrets, or autonomous correction.
```
